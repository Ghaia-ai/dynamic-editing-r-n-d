## correction from v1

v1 of this memo assumed the problem pdfs were third-party uploads authored in illustrator / canva / indesign. that framing was wrong. the majority of pdfs in scope are **generated by `npc-pr-agent` itself**. the bug arises from inconsistency *within* our own pipeline, not from accommodating external files. this rewrite replaces the earlier framing.

## three provenance classes, three failure modes

| class | how the pdf gets made | text layer | editable today? |
|---|---|---|---|
| (a) html → playwright `page.pdf()` | `rendering_service.py:1004, 1062` emits pdf with text via playwright | yes, selectable | partially -- if field map survives handoff |
| (b) png → `_png_to_pdf` embed | `orchestrator.py:1668` embeds a rendered png into a single-page pdf | no, rasterised image | no |
| (c) externally authored, uploaded | third-party pdf in a fill_poster flow | depends on source | only if an admin manually curated a `PDFFieldDefinition` list |

class (a) is what the codebase *tries* to produce. a comment at `rendering_service.py:1054-1058` spells out the intent: *"exported pdf retains selectable/extractable text. this is critical for upgraded posters that are later re-used as pdf templates -- the field-discovery pipeline needs real text spans, not a rasterised image."* the team already knew rasterised output breaks field discovery.

class (b) exists because several export paths (`orchestrator.py:984, 1668, 1943, 2526, 2842, 3056, 3554`) still go via png embedding rather than the html-native pdf path. these produce pdfs that are visually identical to class (a) output but are useless for any downstream text-based editor. **this is an architectural regression hidden inside normal-looking export code.**

class (c) is the original brief framing. much rarer than i originally assumed but not zero: the `fill_poster` tool explicitly supports pre-registered third-party pdfs, and minhal's sample pdfs (`datasets/samples/`) look more like class (c) than class (a) or (b).

## where the editor breaks

```
src/workflows/fill_poster/tools.py :: fill_pdf_template(replacements)
  ↓  reads context.pdf_template_id → cosmos → PDFTemplate
  ↓  .fields: list[PDFFieldDefinition] ← each carries search_pattern (required)
  ↓
src/services/pdf/pdf_editor.py :: PDFEditor.apply_field_update(...)
  ↓  literal-text search using search_pattern
  ↓  cover rect + insert_text at detected position
```

on a class (a) pdf, `search_pattern` works only if the generator produced stable, findable text. in practice the text gets fragmented into multiple glyph runs during layout, and `search_pattern` misses often enough to show up in the `not_found` bucket at `tools.py:236`.

on a class (b) pdf, there is no text to search at all. `search_pattern` returns zero hits unconditionally. the only way to edit these is to roundtrip through the original html source -- which we may or may not still have cached.

on a class (c) pdf, the failure modes i previously listed apply: glyph-run fragmentation, subsetted fonts under obfuscated names, visual-order arabic storage, outlined-text paths.

## the two parallel "templated content" systems

(unchanged from v1; still correct.)

1.  **html template path** (works well)
    -   admin or auto-pipeline uploads raw html.
    -   `template_analyzer.auto_templatize(html)` uses an llm to find dynamic text, converts hardcoded values into `{{ jinja }}` placeholders. returns `(templatized_html, placeholders)`.
    -   end users fill placeholders; `template_filler` + `rendering_service` render.
    -   works because html is already structured -- the dom gives us semantic anchors.

2.  **pdf template path** (the bug domain)
    -   requires a pre-curated `PDFFieldDefinition` list per pdf.
    -   relies on `search_pattern` literal text as the replacement anchor.
    -   fails across all three provenance classes for the reasons above.

## restated problem (v2)

the R&D is actually **two problems in one**:

### problem p1: field-map survival at generation time (class a + b)

when `npc-pr-agent` generates a pdf from html, it *already has* the field map -- the jinja placeholders live in the html source at render time. **we are throwing this information away** at the pdf boundary. the fix on the generation side is to persist the placeholder-to-bbox mapping during render and attach it to the pdf (as a sidecar, as pdf metadata, or as acroform fields) so the editor has an authoritative bbox list without having to rediscover it.

### problem p2: field-map synthesis at edit time (class c, and as a fallback for a/b when p1 fails)

for uploaded pdfs, and as a safety net for our own pdfs when the sidecar is missing or the source html is gone, we need the ability to synthesise a `PDFFieldDefinition` list from pdf bytes alone. this is the problem the four candidate approaches in the brief address.

**solving p1 alone is not enough.** even if every future generation writes a sidecar, the existing library of pdfs in blob storage predates the change, and user-uploaded pdfs will always exist. **solving p2 alone is wasteful.** for class-(a) pdfs we should be preserving the structure we already have, not regenerating it.

the recommended direction is therefore **do both**, with p1 as the cheap first fix and p2 as the durable capability.

## implications for approach selection (v2)

1.  **"extract to structured + regenerate"** (approach a in the brief): the correct tool for **problem p2 on class (a) and (c)** pdfs. reads existing text geometry and synthesises a field list.

2.  **"pdf → html → pdf"** (approach b): attractive for class (b) *only if* we have no cached source html. otherwise we already have the html, and the roundtrip is a fidelity tax with no upside. for class (c), a real candidate.

3.  **"overlay"** (approach c): the correct tool for the *edit* step on any class once a bbox is known. given the generation pipeline already lives in html + playwright, approach c is effectively "do the edit in-place on the pdf as a one-way operation, don't roundtrip."

4.  **"ai-assisted layout understanding"** (approach d): primitive for any approach, not a standalone editor. the deep-research report (`./approach-d-layout-ai.md`) nominates azure document intelligence as the detector of record with gemini 2.5 pro as a fallback.

## restated candidates (post-v2 analysis)

| # | approach | fits problem | class coverage | primary risk |
|---|---|---|---|---|
| p1 | sidecar at generation | p1 | class (a), class (b) if we re-render | low risk; blast radius is in `rendering_service` |
| a | extract → fielddef → existing editor | p2 | class (a), class (c) | glyph fragmentation on heavily designed pdfs |
| b | pdf → html → html template path → pdf | p2 | class (c) mostly | round-trip visual fidelity, licensing on pdf2htmlex |
| c | overlay (bbox-anchored redact + insert) | edit step for any class | all | font coverage on arbitrary uploads, arabic shaping |
| d | layout-ai detection (azure doc intel primary) | detector primitive | all | arabic reading order, per-page cost |

## open questions carried into phase 3

-   does every class-(a) path actually produce a text-bearing pdf, or are some accidentally rasterised downstream? an audit of the eight `orchestrator.py` pdf-export sites (lines 984, 1668, 1943, 2526, 2842, 3056, 3554 plus the 990 area) would answer this. out of scope for this R&D; valuable in the integration plan.
-   for class (b) pdfs, is the source html still retrievable from the session / blob store? if yes, the "fix" is to regenerate from source rather than edit the rasterised output.
-   what fraction of live `fill_poster` traffic is class (a) vs (b) vs (c)? this shapes which problem we prioritise. a quick grep of recent `PDFTemplate.source` values would tell us.
-   for approach c, the deep-research report points us at azure document intelligence as the detector. arabic reading-order behaviour is documented; needs verification on a real arabic poster (pending sample from minhal's team).
